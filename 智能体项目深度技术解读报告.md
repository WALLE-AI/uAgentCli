## 0. 摘要与项目总览

### 0.1 四项目定位

| 项目 | 语言/运行时 | UI | LLM 接入 | 定位 | 规模 |
|---|---|---|---|---|---|
| **claude-code-main** | TypeScript(strict) / **Bun** | React + **Ink** | Anthropic SDK（可 fallback） | Anthropic 官方 Claude Code CLI（2026-03-31 泄露源码） | ~1,900 文件 / 51 万行 |

四者代表**四种截然不同的架构哲学**：claude-code 是"大而全"的工业级单体，hermes 是"活的"自进化个人助理，opencode 是"服务化"的 Effect-DI 架构，pi 是"小而可扩展"的极简内核。

## 1. 总体架构对比

### 1.1 四种架构范式

**① claude-code —— 单体 + 编译期 feature flag 裁剪**
核心是一个 `Tool` 接口（`src/Tool.ts:362`）撑起全系统；所有能力挂在其上。子系统用 Bun 的 `feature()`（`bun:bundle`）在**编译期**做死代码消除（flag：`PROACTIVE`、`KAIROS`、`AGENT_TRIGGERS`、`COORDINATOR_MODE`、`BRIDGE_MODE`、`TOKEN_BUDGET`…）。优点是运行时零开销、单二进制内聚；代价是代码体量巨大（`QueryEngine.ts` ~46K 行）。

**② opencode —— Effect 服务/层 + Client/Server**
整个系统用 **Effect**（`Context.Service` / `Layer.effect` / `LayerNode.make`）写成显式依赖注入图；Agent 内核跑在 HTTP+WebSocket 服务端（`src/server/server.ts`），所有前端（CLI/TUI/ACP/Desktop/Web）都是生成式 SDK 的客户端。优点是可测试、可分布式、多端统一；代价是 Effect 学习曲线陡峭、抽象层厚。

**③ hermes —— Python 单体 `AIAgent` + 子系统插拔**
一个 `AIAgent` 对象（`run_agent.py:393`）持有 `MemoryManager`、`ContextEngine`、tool registry、transports 等可插拔子系统；conversation loop 被抽出到 `agent/conversation_loop.py`。优点是子系统边界清晰、易替换（记忆/上下文引擎均为插件）；代价是主对象巨大（`cli.py` ~18K 行）。

**④ pi —— 4 包 monorepo + 事件/hook 扩展**
`pi-ai`（LLM）/ `pi-agent-core`（运行时）/ `pi-coding-agent`（CLI）/ `pi-tui`（UI）分层清晰；内核只保留 7 个工具，其余能力（沙盒、渠道、编排）全部通过庞大的 `ExtensionAPI`（`extensions/types.ts`）以第三方 TypeScript 模块 hook 进来。优点是内核小、可审计、可扩展；代价是"开箱即用"能力弱，需要拼装。

### 1.2 对 uAgentCli 的启示

- **不要照抄 claude-code 的 monolith 体量**，但要学它的 `Tool` 接口作为统一抽象骨架。
- **借鉴 opencode 的 Client/Server + SDK** 作为多端接入的地基（CLI 只是其中一个客户端）。
- **借鉴 hermes 的"子系统插拔"**：记忆、上下文、渠道、沙盒都定义为接口/Provider，默认实现 + 可替换。
- **借鉴 pi 的 hook 扩展模型** 让高级能力可选、可外置。

## 2. agent-run-loop（智能体主循环）

### 2.1 原理与关键实现

**claude-code** —— `query()`（`src/query.ts:219`，`async function*` 生成器）→ `queryLoop()`（`query.ts:241`）。每轮 `while(true)`：
1. **前置上下文削减**（顺序）：tool-result budget → snip → microcompact → context-collapse → autocompact（`query.ts:379-543`）；
2. **流式 API 调用** `deps.callModel(...)`；关键设计：`stop_reason==='tool_use'` 被视为**不可靠**，改用流式中出现 `tool_use` block 置 `needsFollowUp=true` 作为唯一续轮信号（`query.ts:826-845`）；
3. **边流边执行**：`StreamingToolExecutor` 在模型还在流式输出时就开始跑工具（`query.ts:838-862`）；
4. **多级错误恢复**（最大亮点）：流式期间错误对 SDK 消费者**隐藏**直到恢复尝试完成 —— model fallback（`FallbackTriggeredError`）、prompt-too-long → context-collapse → reactive compact、max_output_tokens 8k→64k 升级再多轮恢复（`query.ts:1085-1256`）。
工具分派 `runTools()`（`toolOrchestration.ts:19`）：连续的**并发安全**（只读）工具并行跑（默认并发 10），非安全工具串行。子智能体：`AgentTool` 递归调用 `query()`，隔离 `agentId` 与文件状态缓存。

**hermes** —— `run_conversation()`（`agent/conversation_loop.py:523`，已核实）。前置 prologue 由 `build_turn_context()` 处理（消息清洗、todo/nudge 注入、系统提示 restore-or-build、preflight 压缩、external-memory 预取）。工具分派 `agent/tool_executor.py` 的 `execute_tool_calls_concurrent()`（`:325`）/ `sequential()`（`:1022`），并行与否由 `_should_parallelize_tool_batch()` 决定，带**三重安全门**：破坏性命令检测、MCP 并行安全、**文件路径重叠检测**（`_paths_overlap`）避免并发工具改到重叠路径。终止时有 **verify-on-stop**：注入合成 "done" + "nudge" 逼智能体再迭代一轮。子智能体：`delegate_task`（`tools/delegate_tool.py`，深度上限 `MAX_DEPTH=1`）。还支持 **MoA（Mixture-of-Agents）**（`agent/moa_loop.py`）。

**opencode** —— `SessionPrompt.runLoop`（`src/session/prompt.ts:1081`，已核实，Effect 生成器）。终止条件：最后一条 assistant 消息 finish reason 非 `tool-calls`、无 pending 工具、且晚于最后 user 消息（`prompt.ts:1111-1130`）。逐轮由 `SessionProcessor.process`（`processor.ts:627`）消费 `Stream<LLMEvent>`；**工具实际在 AI SDK 内部执行**，processor 只记录 `pending→running→completed` 状态并回填。亮点：**死循环守卫** `DOOM_LOOP_THRESHOLD=3`（连续 3 次相同 tool+input 触发权限 ask，`processor.ts:356-380`）+ Effect 重试策略。

**pi** —— `runLoop()`（`packages/agent/src/agent-loop.ts:155`，已核实）+ `AgentHarness`。特色是 **三队列可插队 steering**：`steerQueue` / `followUpQueue` / `nextTurnQueue`（`agent-harness.ts:176-180`），对外暴露 `prompt()/steer()/followUp()/nextTurn()/abort()`。`"length"` 停止会把所有工具调用标记为可能截断并失败。子智能体：**loop 内无 Task 工具**，多智能体靠外置 orchestrator RPC 子进程。

### 2.2 对比表

| 维度 | claude-code | hermes | opencode | pi |
|---|---|---|---|---|
| 循环形态 | async 生成器 + 可变 State | Python 函数循环 + 计数器 | Effect 生成器 | 内外双层循环 |
| 续轮信号 | 流式 `tool_use` block（弃用 stop_reason） | assistant 含 tool_calls | finish reason ≠ tool-calls | hasMoreToolCalls / pending |
| 工具执行 | **边流边执行** StreamingToolExecutor | 并发/串行 + 路径重叠检测 | AI SDK 内部执行 | 并行/串行按 executionMode |
| 错误恢复 | ✅ **多级**（fallback/压缩/token 升级） | length/truncated 重试 + verify | Effect 重试策略 | length 截断失败处理 |
| 死循环防护 | — | 迭代预算 iteration_budget | ✅ DOOM_LOOP=3 | — |
| 插队/steering | 命令队列优先级 | 中断-重定向 | 队列 | ✅ **三队列** |
| 子智能体 | ✅ AgentTool 递归 + coordinator | ✅ delegate_task(深度1)+MoA | ✅ task 工具 | ⚠️ 外置 orchestrator |

### 2.3 优劣分析

- **claude-code 最强在"韧性"**：多级恢复 + 边流边执行让长任务几乎不会因单点错误/超长上下文中断，代价是逻辑极复杂（`query.ts` 1730 行）。
- **hermes 最强在"并发安全"**：路径重叠检测是其它三家没有的，避免并行文件工具互相踩踏；verify-on-stop 提升任务完成度。
- **opencode 胜在"简洁可靠"**：把工具执行下沉给 AI SDK，自己只做状态机 + 死循环守卫，工程量小。
- **pi 胜在"交互性"**：三队列 steering 让用户可随时插话/改向，是最好的人机协作循环。

### 2.4 对 uAgentCli 的启示

主循环建议：**opencode 的清晰状态机 + claude-code 的多级恢复 + hermes 的并发路径检测 + pi 的 steering 队列**。续轮信号不要依赖 `stop_reason`，用是否产生 tool_use 判定更稳。务必加 DOOM_LOOP 守卫与迭代预算。

### 2.5 uAgentCli 落地细化设计

> 目标：一个 **可测试的纯状态机内核** + **可外挂的恢复/守卫策略**。内核不直接跑工具，只产出事件；工具执行、恢复、steering 都通过接口注入，便于单测与替换。

**① 核心接口（TypeScript，内核为 async 生成器）**

```ts
// packages/core/src/loop/agent-loop.ts
export type LoopEvent =
  | { type: 'text.delta'; text: string }
  | { type: 'tool.pending'; call: ToolCall }        // 模型流式产出 tool_use，尚未执行
  | { type: 'tool.running'; id: string }
  | { type: 'tool.result'; id: string; result: ToolResult }
  | { type: 'turn.end'; reason: 'complete' | 'tool-calls' | 'length' | 'aborted' }
  | { type: 'recovery'; kind: RecoveryKind; attempt: number }   // 恢复对上层可见但不中断

export interface AgentLoop {
  run(state: LoopState): AsyncGenerator<LoopEvent, LoopState>
}

export interface LoopState {
  messages: Message[]
  iteration: number
  iterationBudget: number          // 硬上限，防失控（借 hermes iteration_budget）
  recentToolSigs: string[]         // DOOM_LOOP 守卫用（借 opencode）
}
```

**④ DOOM_LOOP 守卫 + 恢复阶梯（借 opencode + claude-code）**
```ts
const DOOM_LOOP_THRESHOLD = 3
function isDoomLoop(state: LoopState, call: ToolCall) {
  const sig = `${call.name}:${stableHash(call.input)}`
  const n = state.recentToolSigs.filter(s => s === sig).length
  return n >= DOOM_LOOP_THRESHOLD   // 连续 3 次同 tool+input → 强制转 permission ask
}

// 恢复阶梯（错误对上层隐藏，直到本级恢复失败才冒泡）：
// L1 model fallback → L2 prompt-too-long: context-collapse → L3 reactive compact
// L4 max_output_tokens 升级(8k→64k) → L5 冒泡为 turn.end{aborted}
type RecoveryKind = 'model-fallback' | 'context-collapse' | 'reactive-compact' | 'token-upgrade'
```

**⑤ 三队列 steering（借 pi agent-harness.ts:176）**
```ts
class AgentHarness {
  private steerQueue: string[] = []      // 立即插队（打断当前轮方向）
  private followUpQueue: string[] = []   // 当前轮结束后追加
  private nextTurnQueue: string[] = []   // 下一轮开头注入
  prompt(msg) {} steer(msg) {} followUp(msg) {} nextTurn(msg) {} abort() {}
}
```

**⑥ 落地取舍**：内核只依赖 `callModel`、`runTools`、`recover`、`compact` 四个注入接口 → 100% 可单测、可 mock。子智能体先用 **AgentTool 递归**（借 cc）满足 MVP；分布式 orchestrator（借 pi）留作阶段 4。**MVP 必带**：续轮=tool_use、DOOM 守卫、iterationBudget、L1+L3 恢复；**后置**：边流边执行、L2/L4、MoA。

## 3. Tools（工具集）

### 3.1 原理与关键实现

**注册机制四种范式**：

- **claude-code**：`buildTool(def)`（`Tool.ts:783`）铺 `TOOL_DEFAULTS`（**fail-closed**：`isConcurrencySafe→false`、`isReadOnly→false`、`checkPermissions→allow`）。单一真源 `getAllBaseTools()`（`tools.ts:193`）静态数组 + feature flag 条件展开；`getTools(ctx)` 按 deny 规则 + `isEnabled()` 过滤；`assembleToolPool()` 合并 MCP 工具时分区各自排序以**保持 prompt-cache 稳定**。~60 个工具目录。
- **opencode**：`Tool.define(id, initEffect)`，参数用 **Effect Schema**；`wrap()` 解码校验 + 自动截断输出。注册来源：内置硬列表 + 文件系统自定义（glob `{tool,tools}/*.{js,ts}` 动态 import）+ 插件 + MCP。
- **hermes**：**导入即注册** —— 每个 tool 模块在 import 时调 `registry.register(name, toolset, schema, handler, check_fn, requires_env, ...)`（`tools/registry.py:365`）；`discover_builtin_tools()`（`:70`）**AST 扫描** `tools/*.py` 找顶层 `registry.register(...)` 调用，只 import 真正注册工具的模块（省启动、防循环导入）。
- **pi**：TypeBox schema + 工厂函数（`createXTool` + `createXToolDefinition`）；**内核仅 7 个工具**（read/bash/edit/write/grep/find/ls），其余靠扩展 `pi.registerTool(...)` 补齐。

**工具集分组（toolset）**：hermes 最系统 —— `toolsets.py` 的 `TOOLSETS` dict（`:95`）把命名 bundle（web/search/vision/terminal/safe/moa…）映射到工具列表 + `includes` 传递组合；且大量工具用 **`check_fn` 门控**（如 `computer_use` 仅 macOS 装了 cua-driver 才进 schema、`kanban_*` 仅 kanban worker 才可见）。claude-code 用 feature flag + deny/allow 列表达到类似效果。

**大工具集应对**：claude-code 的 `ToolSearchTool`（延迟工具 `shouldDefer:true`，用前需关键词搜索）；hermes 的 `managed_tool_gateway` + `tool_search`。

**MCP**：四家均把 MCP 工具并入同一工具池 + 权限管线；claude-code/opencode 支持 stdio/SSE/Streamable HTTP + OAuth 最完整。

### 3.2 对比表

| 维度 | claude-code | hermes | opencode | pi |
|---|---|---|---|---|
| Schema | Zod v4（lazy） | OpenAI JSON schema | Effect Schema | TypeBox |
| 注册 | 静态数组 + flag | **import 即注册 + AST 发现** | define + 文件/插件/MCP | 工厂函数 + 扩展 |
| 默认安全姿态 | ✅ fail-closed | check_fn 门控 | permission 派生 | — |
| 工具集分组 | flag/allow/deny | ✅ **toolset + check_fn** | agent ruleset | 扩展 bundle |
| 大工具集 | ✅ ToolSearch 延迟 | ✅ managed gateway | 权限可见性过滤 | — |
| 内置工具数 | ~60 | ~100 | ~15 | **7** + 扩展 |

### 3.3 优劣分析

## 5. 上下文管理（Context Management）

### 5.1 原理与关键实现

- **claude-code**：**5 级压缩流水线**（每轮迭代按序执行，逐级喂下一级，`query.ts:379-648`）：① tool-result budget（超限结果落盘换文件预览）→ ② snip（`HISTORY_SNIP` 删陈旧工具输出）→ ③ microcompact（按 `tool_use_id` 外科式丢旧结果，`CACHED_MICROCOMPACT` 编辑 prompt cache）→ ④ context-collapse（对全历史的读时投影 + 持久 commit log，保留粒度而非摘要）→ ⑤ autocompact（**fork 出 compact 子智能体**做全量摘要）。另有 `task_budget`（beta `task-budgets-2026-03-13`）跨压缩边界追踪。静态上下文 `getSystemContext`/`getUserContext` memoized 每轮前置。
- **hermes**：**可插拔 `ContextEngine`**（`agent/context_engine.py` 抽象，config `context.engine` 默认 `compressor`，可放 `plugins/context_engine/<name>/`）。默认 `ContextCompressor`（`agent/context_compressor.py`，3342 行）：`threshold_percent=0.75` 触发压缩，`protect_first_n=3`（头）+ `protect_last_n=6`（尾）保护，中间摘要；**preflight 压缩**在模型调用前先跑防溢出。
- **opencode**：`SessionCompaction.process`（`compaction.ts:289`）—— 保留 tail（`preserve_recent_tokens` 默认 25% usable，clamp 2k–8k）摘要 head，链式串接历史摘要只回放"最新摘要 + tail"；工具输出 cap `TOOL_OUTPUT_MAX_CHARS=2000`。独立的 `prune()`（`:243`）在循环结束后 GC 老工具输出（`PRUNE_PROTECT=40k`，保护 skill 输出）。
- **pi**：**独特的会话树（branch）** 而非线性历史（`harness/session/`）。`shouldCompact()` 在 `contextTokens > contextWindow - reserveTokens`（默认 reserve 16384、keepRecent 20000）时触发；`findCutPoint` 在有效消息边界切；`navigateTree()` + `branch-summarization.ts` 跳分支时摘要分支。

### 5.2 对比表

| 维度 | claude-code | hermes | opencode | pi |
|---|---|---|---|---|
| 触发 | 多阈值分级 | 0.75 阈值 | overflow 检测 | window-reserve |
| 策略 | ✅ **5 级流水线** | head/tail 保护摘要 | 保留 tail + 链式摘要 | cut point + 分支摘要 |
| 工具输出 GC | ✅ budget+snip+micro | — | ✅ prune 40k | — |
| 历史模型 | 线性 + collapse 投影 | 线性 | 线性 + 摘要链 | ✅ **会话树 branch** |
| 可插拔 | flag 控制 | ✅ **ContextEngine 插件** | 插件 hook 覆盖 | — |
| 增量摘要 | ✅ microcompact | — | ✅ 链式 | ✅ 分支摘要 |

### 5.3 优劣分析

- **claude-code 的 5 级流水线** 最精细：能在"不摘要"（microcompact/collapse 保粒度）与"摘要"之间平滑过渡，最大化保真度，但复杂度极高。
- **hermes 的可插拔 ContextEngine** 工程上最灵活：换压缩策略不动主循环。
- **opencode 的 tail+prune** 简单有效，prune 单独 GC 是好设计。
- **pi 的会话树** 独树一帜：适合探索式任务（分支尝试 + 回溯），但线性对话收益不明显。

### 5.4 对 uAgentCli 的启示

采用 **hermes 式可插拔 ContextEngine 接口**，默认实现走 **claude-code 的分层思路**（先 tool-result budget → microcompact 保粒度 → 最后才 autocompact 摘要）+ **opencode 的 prune GC**。会话树可作为可选高级特性。

### 5.5 uAgentCli 落地细化设计

> 目标：**可插拔 ContextEngine 接口**（借 hermes），默认实现走 **保真度优先的分层流水线**（借 cc），换策略不动主循环。

**① 接口（借 hermes ContextEngine 抽象，config `context.engine`）**
```ts

  // 每轮模型调用前跑，防溢出（借 hermes preflight）
  prepare(messages: Message[], budget: TokenBudget): Promise<Message[]>

```

**② 默认分层流水线（借 cc 5 级，按序逐级喂下一级）**

```
② snip               : 删陈旧工具输出（HISTORY_SNIP）
③ microcompact       : 按 tool_use_id 外科式丢旧结果【保粒度、不摘要】，编辑 prompt cache
④ context-collapse   : 全历史读时投影 + 持久 commit log（保粒度而非摘要）
```

关键阈值（借各家）：`threshold=0.75` 触发（hermes）；`protect_first_n=3 + protect_last_n=6`（hermes）或 `preserve_recent 25% clamp 2k–8k`（opencode）；`TOOL_OUTPUT_MAX_CHARS=2000`（opencode）。

**③ prune GC（借 opencode compaction.ts:243，独立于压缩）**
```ts

prune(messages) { /* 保留近 40k token 内 + skill 结果，其余 tool 输出置换为占位 */ }
```

**④ 会话树（可选高级特性，借 pi harness/session）**：探索式任务用 branch 而非线性历史；`navigateTree()` 跳分支时 `branch-summarization` 摘要分支。**线性对话收益不明显，列为阶段 4 可选**。

**⑤ 落地取舍**：**MVP** = ContextEngine 接口 + `threshold` 触发 + head/tail 保护 + 一次性 autocompact 摘要（对齐 opencode/hermes 可用水平）；**阶段 2** = microcompact 保粒度 + prune GC；**阶段 4** = context-collapse + 会话树。保真度原则：**能"不摘要"就不摘要**（先 budget/microcompact 保粒度），摘要是最后一级。

## 6. 记忆管理（Memory Management）

### 6.1 原理与关键实现

**重要行业信号：四家均不使用向量库/embeddings 做长期记忆**，一律走"文件 + 全文检索"。

- **claude-code（最强）**：**两套记忆并存**。① CLAUDE.md 项目指令（`utils/claudemd.ts` 注入 `getUserContext`，嵌套 CLAUDE.md 随目录触碰以 `nested_memory` 附加）；② **类型化持久 memdir**（`~/.claude/projects/<slug>/memory/`）：`MEMORY.md` 是 200 行/25KB 上限的**索引**（始终入 context），实际记忆存分主题 `.md` + frontmatter（`type: user|feedback|project|reference`）；两步保存协议（写主题文件 + 往 MEMORY.md 加一行指针）。召回：`startRelevantMemoryPrefetch`（`query.ts:301`）每轮后台预取相关记忆。还有自动抽取（`extractMemories`）、**team memory 同步**、KAIROS 每日日志 + nightly `/dream` 蒸馏。
- **hermes（强）**：`MemoryManager`（`agent/memory_manager.py`）委托 `MemoryProvider`（同一时刻只允许一个外部 provider 防 schema 膨胀）。生命周期：`prefetch_all(user_message)` 预取召回 + `queue_prefetch_all()` 异步预热下一轮 + `sync_all(user, assistant)` 事后持久化。**周期性 nudge**（`turn_context.py:285`）每 N 轮提示智能体持久化知识，计数器跨会话从历史 hydrate。**跨会话检索**：`session_search_tool.py` 走 **SQLite FTS5** 索引（三模式：DISCOVERY 全文→去重会话±5 消息窗、SCROLL 锚定窗、BROWSE 近期），零 LLM 成本。另有学习图 `learning_graph.py` + Honcho 用户建模。
- **opencode（弱）**：仅 `Instruction`（`src/session/instruction.ts`）注入 AGENTS.md/CLAUDE.md/CONTEXT.md（含全局 + http URL），`resolve()` 从模型 read 的文件向上找嵌套 AGENTS.md 附加（每消息去重）。**无向量、无跨会话语义召回**，session 存 SQLite。
- **pi（弱）**：`loadProjectContextFiles()`（`resource-loader.ts:85`）从 cwd 向上收 AGENTS.md/CLAUDE.md 注入 `<project_context>`；session 存 JSONL 树可 resume/fork。`memory-repo.ts` 只是**内存会话存储**非长期记忆。**该维度基本缺失**。

- **全行业不用向量库** 说明：对 Agent 记忆，结构化文件 + 关键词/全文检索的**可解释性、可编辑性、低成本**目前压过语义向量的召回率优势。

### 6.4 对 uAgentCli 的启示

记忆栈建议：**claude-code memdir（类型化文件 + MEMORY.md 索引）作主存 + hermes FTS5 作跨会话检索 + 周期 nudge 主动持久化**。向量语义召回作为**可选后置**增强，不作默认。

### 6.5 uAgentCli 落地细化设计

> 目标：**文件为主存（可读可编辑）+ 全文检索（零 LLM 成本召回）+ 主动持久化**。全行业不用向量库，uAgentCli 亦以文件+FTS 为默认。

## 7. 渠道管理（Channel Management）

### 7.1 原理与关键实现

- **hermes（最强）**：**单 Gateway 进程驱动 ~28 平台**。入口 `gateway/run.py` `start_gateway()`。适配器基类 `gateway/platforms/base.py`（统一 `send_message`/收发、按平台归一化音频格式）。注册中心 `gateway/platform_registry.py`（`PlatformEntry`：名称/工厂/可用性检查/必需 env/cron `deliver=` 目标钩子/YAML→env 翻译），用**懒加载** `register_deferred` 让 ~20 个重 SDK 按需 import。渠道：telegram/discord/slack/whatsapp/matrix/teams/feishu/dingtalk/wecom/line/google_chat/mattermost/irc/sms/email/ntfy…（`plugins/platforms/`）+ signal/weixin/bluebubbles(iMessage)/qqbot/msgraph（`gateway/platforms/`）+ HTTP api_server。内部：`session.py`（每会话状态）、`stream_consumer/dispatch`（流式外发）、`delivery.py`、`mirror.py`（**跨渠道镜像/会话连续性**）、`pairing.py`（身份链接）、`relay/`（WS 中继）。语音备忘录入站转写。
- **claude-code**：`cli.tsx` 顶层分发到多种传输：交互 TUI（Ink）/ headless `--print`（**NDJSON** 结构化 IO）/ **IDE Bridge**（`BRIDGE_MODE`，VS Code/JetBrains，JWT 认证）/ remote WebSocket / Direct Connect server / **MCP 作为审批渠道**（Telegram/iMessage 上回 "yes abc123" 批准权限，`interactiveHandler.ts:316`）。
- **opencode**：**Client/Server 为本**。Effect `HttpApi` + WebSocket + SSE + **mDNS** 发现（`src/server/server.ts`）；事件总线（`src/bus/`）向客户端推 SSE/WS。客户端：CLI `run`（`--format json`）/ TUI（可 `--attach` 已运行 server）/ **ACP**（Agent Client Protocol，编辑器集成）/ Desktop(Electron)/Web/GitHub/Slack bot。统一走生成式 SDK（`createOpencodeClient`）。
- **pi**：三模式（`packages/coding-agent/src/modes/`）：Interactive(TUI) / Print(一次性) / **RPC**（newline-delimited JSON-RPC over stdio，`RpcClient` spawn `pi` 子进程）。扩展可通过 `sendMessage`/`ExtensionUIContext` 注入 IO。**聊天平台外置**（另一仓库 `pi-chat`）。

### 7.2 对比表

|---|---|---|---|---|
| 架构 | 多传输分发 | ✅ **单 Gateway 多适配器** | ✅ Client/Server+SDK | 三运行模式 |
| 聊天平台数 | MCP 间接 | ✅✅ **~28 内置** | Slack/GitHub | ❌ 外置 pi-chat |

| IDE 集成 | ✅ Bridge(JWT) | — | ✅ ACP | — |
| 服务发现 | — | — | ✅ mDNS | — |
| 会话连续性 | 远程 session | ✅ mirror 跨渠道 | server 集中 | — |
| 审批渠道化 | ✅ MCP 手机批准 | ✅ gateway async | ✅ 权限 API | 扩展 UI |

### 7.3 优劣分析

- **hermes 是 IM/聊天多渠道之王**：单进程驱动近 30 平台 + 懒加载 + 跨渠道镜像，"从 Telegram 指挥云端 VM 上干活"是其定位核心。

## 8. 心跳技术（定时任务 + 主动触发）

> 用户重点关注："定时任务"与"根据监控/记忆/任务主动执行"。**这是四家差异最大的维度** —— 只有 claude-code 与 hermes 真正具备主动性，opencode/pi 缺失。

### 8.1 原理与关键实现

- **claude-code（强）**：**Durable cron** —— `CronCreateTool`（`CronCreateTool.ts:56`）接受 5 字段 cron + prompt + `recurring` + `durable`；durable 任务持久化到 `.claude/scheduled_tasks.json` 跨重启存活（`MAX_JOBS=50`，自动过期）。心跳本体 `useScheduledTasks` hook（`useScheduledTasks.ts:40`）挂 `createCronScheduler`（timer + 文件监视 + jitter），触发时**注入 user 消息给 teammate** 或**以 `'later'` 优先级入命令队列**（`WORKLOAD_CRON` 低 QoS，"没人在等"）。**主动触发**：`SleepTool`（proactive 等待）、`MonitorTool`（监控条件）、`RemoteTriggerTool`、`SubscribePRTool`（GitHub webhook → user 消息）、`daemon` 长驻模式、`autoDream` nightly 记忆整合。
- **hermes（强）**：`cron/scheduler.py` `tick()` 由 **gateway 每 60 秒**从后台线程调用，用 **fcntl 文件锁**（`~/.hermes/cron/.tick.lock`）防重复运行；任务支持 cron 表达式/间隔（croniter）、**自然语言**、**投递任意平台**（`deliver=`）；`tools/cronjob_tools.py` 让智能体**自己创建/编辑 cron**。**非 cron 主动触发**：**Curator 闲时触发**（idle 且距上次 > interval_hours 就 spawn 后台审校 agent）、memory/skill **turn-count nudge**（`turn_finalizer.py:486`）、`kanban_watchers.py` / `memory_monitor.py` 后台监控。serverless（Modal/Daytona）hibernate + wake-on-message。
- **opencode（缺）**：**无 cron/timer/定时唤醒**。仅进程内 `background-job.ts`（内存 job 注册表，**不持久**，重启即丢，`:113-119` 明确注明）+ 后台子智能体完成时**异步回调**注入结果。要定时得自建。
- **pi（缺）**：**无 cron、无主动唤醒**。仅 orchestrator `radius.ts` 的 **presence 心跳**（`scheduleMachineHeartbeat`/`schedulePiHeartbeat`，`setTimeout` 自重排 + 指数退避）—— 这是**分布式 liveness 心跳**（向 radius.pi.dev 报活），非"主动执行任务"。UI 倒计时是装饰。

### 8.2 对比表

| 维度 | claude-code | hermes | opencode | pi |
|---|---|---|---|---|
| 定时 cron | ✅ 5 字段 + **durable 持久** | ✅ 60s tick + **文件锁防重** | ❌ | ❌ |
| 智能体自建任务 | ✅ CronCreateTool | ✅ cronjob_tools | — | — |
| 结果投递 | teammate/队列 | ✅ **任意渠道 deliver** | — | — |
| 监控触发 | ✅ MonitorTool | ✅ kanban/memory watcher | — | — |
| 记忆/任务触发 | autoDream nightly | ✅ **闲时 Curator + nudge** | — | — |
| webhook 触发 | ✅ SubscribePR | gateway 入站 | — | — |
| 后台 job | 命令队列 | 后台线程 | ⚠️ 内存不持久 | — |
| 分布式心跳 | 远程 session | serverless wake | — | ⚠️ Radius presence |

### 8.3 优劣分析

- **claude-code 与 hermes 是仅有的两个"能主动干活"的智能体**：cron 定时 + 监控/记忆/闲时触发俱全。
- **claude-code 的 durable cron + jitter + 低 QoS 队列** 工程最完备；**hermes 的文件锁防重 + 自然语言任务 + 任意渠道投递 + Curator 闲时自维护** 最贴近"7×24 个人助理"。
- **opencode 的 background-job 不持久** 是明显短板（作者已注明），无定时。
- **pi 的 Radius** 只是报活，与"主动执行"无关。

### 8.4 对 uAgentCli 的启示

若要主动性（用户明确诉求），**必须自建**，直接抄 **claude-code + hermes 的组合**：① durable cron（持久化 JSON + 文件锁防重 + jitter）；② 智能体自建/自编辑 cron 工具；③ 结果投递到任意渠道；④ **闲时/监控/记忆-任务驱动的主动触发**（Curator 式后台审校 + turn-count nudge + watcher）。opencode/pi 在此维度**不可参考**。

## 9. 沙盒技术（Sandbox）

### 9.1 原理与关键实现

- **claude-code（强）**：`sandbox-adapter.ts` 封装外部 `@anthropic-ai/sandbox-runtime`（Linux **bubblewrap** / macOS **Seatbelt**）。`convertToSandboxRuntimeConfig`（`:172`）把 CC 权限规则翻译成沙盒限制（WebFetch domain → 网络 allow/deny；Edit/Read → allowWrite/denyRead）。**逃逸加固（亮点）**：无条件 `denyWrite` 所有 settings.json（防配置逃逸）、`.claude/skills`、**bare-git 文件**（HEAD/objects/refs/hooks/config，防经未沙盒 git 逃逸，`scrubBareGitRepoFiles`）。沙盒时 `autoAllowBashIfSandboxed` 自动放行 Bash。另有 **worktree 隔离**（`EnterWorktreeTool` 在 `.claude/worktrees/` 建隔离 git worktree）。
- **hermes（最强，后端多样性）**：`tools/environments/base.py` `BaseEnvironment` ABC，统一 **spawn-per-call**（每命令 fresh `bash -c`，一次快照 env/functions/aliases 再逐命令 re-source，CWD 靠 stdout marker 持久）。**6 后端**：local/docker/ssh/singularity/**modal**/**daytona**（后两 serverless 休眠唤醒）。**Docker 加固**：`cap-drop ALL` + `no-new-privileges` + PID 限制 + CPU/内存/磁盘限制 + **凭证转发 blocklist**（`_HERMES_PROVIDER_ENV_BLOCKLIST` 防 provider 密钥泄进沙盒）。配 s6 监督 + nix。补充路径/命令隔离：`path_security.py`、`url_safety.py`（SSRF/egress）、`secret_scope.py`+`redact.py`（密钥作用域/脱敏）。
- **opencode（中，静态分析折中）**：**无内核沙盒**。shell 工具（`src/tool/shell.ts`）用 **tree-sitter**（bash+PowerShell WASM）解析命令 AST，`collect()`（`:378`）提取每命令前缀做 `bash` 权限模式 + 检测**触碰文件的命令**（rm/cp/mv/cat…）解析路径参数，任何解析出的路径**在 instance 目录/worktree 之外**触发 `external_directory` 权限 ask。有可选 `packages/containers`。
- **pi（无，完全外置）**：README 明确"以启动用户全权限运行"。bash 走本地 `child_process.spawn`（detached + 进程树 kill）。隔离**外置三方案**：Gondolin micro-VM 扩展（覆盖 `BashOperations` 把工具路由进本地 Linux 微 VM）/ 纯 Docker / OpenShell 策略沙盒。`BashOperations` 接口（`bash.ts:56`）是重定向 seam。

### 9.2 对比表

| 维度 | claude-code | hermes | opencode | pi |
|---|---|---|---|---|
| 内核隔离 | ✅ bubblewrap/Seatbelt | ✅ Docker/容器/远程 | ❌ 静态分析替代 | ❌ 无 |
| 后端数 | 本地沙盒 + worktree | ✅✅ **6 后端** | 可选 containers | 外置 3 方案 |
| serverless 休眠 | — | ✅ Modal/Daytona | — | — |
| 逃逸加固 | ✅✅ settings/bare-git denyWrite | ✅ cap-drop/密钥 blocklist | — | — |
| 命令静态分析 | — | 破坏性检测 | ✅ **tree-sitter AST** | — |
| 路径边界 | 规则翻译 | path_security | ✅ external_directory | — |
| 可插拔后端 | 适配器 | ✅ BaseEnvironment | — | ✅ BashOperations seam |

### 9.3 优劣分析

- **claude-code 的逃逸加固** 是安全深度最好的：想到了 settings.json/bare-git 这类隐蔽逃逸面。
- **hermes 的 6 后端 + serverless** 灵活性最强：本地到云端 VM 到休眠唤醒全覆盖，Docker 加固完备。
- **opencode 的 tree-sitter 静态分析** 是"无沙盒下的最佳努力"：靠解析命令决定权限，折中但无真隔离。
- **pi 完全外置**：内核零负担，但安全全靠用户自己容器化。

### 9.4 对 uAgentCli 的启示

定义 **hermes 式 `BaseEnvironment` 接口 + pi 式 `BashOperations` seam**（后端可插拔）；默认实现给 **bubblewrap（Linux）/ Docker cap-drop 加固**；抄 **claude-code 的逃逸加固**（settings/bare-git denyWrite）；至少上 **opencode 的 tree-sitter 命令静态分析**作为无沙盒时的兜底。

### 9.5 uAgentCli 落地细化设计

> 目标：**后端可插拔的隔离层**（借 hermes/pi）+ **强逃逸加固**（借 cc）+ **无沙盒时的静态分析兜底**（借 opencode）。

**① 可插拔后端接口（借 hermes BaseEnvironment + pi BashOperations seam）**

## 10. 权限/安全管理（Permission & Security）

### 10.1 原理与关键实现

- **claude-code（强）**：权限模式 `default/acceptEdits/plan/bypassPermissions/dontAsk/auto`（`types/permissions.ts:16`）。`ToolPermissionContext`（`Tool.ts:122`）持 allow/deny/ask 规则**按来源分级**（userSettings/projectSettings/policySettings/cliArg/session…）。每工具 `checkPermissions()` 返回 `allow/deny/ask/passthrough`。**审批竞速（亮点）**：`ask` 决策通过原子 `claim()`/`resolveOnce` **多路竞速**（`interactiveHandler.ts:57`）—— 本地对话 / IDE Bridge/CCR 远程 / IM 渠道（Telegram/iMessage）/ PermissionRequest hook / **bash 安全分类器**（可自动放行安全命令），首个响应者胜出。**2 段 LLM 安全分类器**（`auto` 模式，fast→thinking 评估整条 transcript）。**全量 hook 事件体系**（PreToolUse/PostToolUse/UserPromptSubmit/SessionStart/Stop/PreCompact/PermissionRequest…，来自 settings + skill/agent frontmatter），Stop-hook 可阻止终止。SSRF guard 护 HTTP hook。
- **hermes（强）**：`tools/approval.py` 是"危险命令系统单一真源"。**两级 blocklist**：`HARD_BLOCKED`（`:519`，"绝不运行，即便 YOLO"，先于任何 bypass）vs 软 `DANGEROUS_PATTERNS`（`:595`，rm -rf/chmod 777/curl|sh，需审批但 YOLO 可过）。**frozen YOLO**：`HERMES_YOLO_MODE` **import 时冻结**（`_YOLO_MODE_FROZEN`，防进程内 skill/注入改 env 绕过）。**contextvar 会话隔离修 CVE**（GHSA-96vc-wcxf-jjff：原进程全局 `HERMES_INTERACTIVE` 在并发 ACP/gateway 会话间竞态可误批危险命令，改用 contextvars）。**辅助 LLM 智能审批**（自动批低风险）+ 持久 allowlist。威胁/依赖扫描：`threat_patterns.py`/`tirith_security.py`/`osv_check.py`。子智能体默认 `_subagent_auto_deny`。
- **opencode（中）**：规则 `{permission, pattern, action∈allow/ask/deny}`，`evaluate` 取**最后匹配规则**（last-wins，让分层 defaults→user→agent→session 确定性叠加）。`ask` 三答 `once/always/reject`，**`reject` 带消息变 `CorrectedError` 反馈回模型**（拒绝也能引导）。agent 即 ruleset（build 全开、plan 拒所有 edit、explore 几乎全 deny）。plugin hook（`tool.definition`/`shell.env`/`chat.messages.transform`…）是扩展/安全钩子。工具可见性按 deny 隐藏。
- **pi（弱）**：**无 allow/deny 命令权限**（README 明示）。靠 **project trust**（`trust-manager.ts`，`~/.pi/agent/trust.json`，最近祖先匹配；未信任项目跳过加载项目本地 settings/extensions/skills/SYSTEM.md 等**可执行代码**）+ `beforeToolCall` hook（可 block）/`afterToolCall`（可改写/终止）+ 供应链加固（`.npmrc` save-exact/min-release-age、shrinkwrap allowlist）。

### 10.2 对比表

| 维度 | claude-code | hermes | opencode | pi |
|---|---|---|---|---|
| 权限模型 | 6 模式 + 来源分级规则 | 两级 blocklist + 模式 | last-wins 规则 | ❌ 无命令权限 |
| 审批流 | ✅✅ **多路竞速** | CLI+gateway async | once/always/**reject 反馈** | hook block |
| 绕过防护 | classifier 兜底 | ✅✅ **frozen YOLO + HARD_BLOCKED** | deny 优先 | trust 门控 |
| 并发安全 | claim 原子 | ✅ **contextvar 修 CVE** | Effect 隔离 | — |
| LLM 辅助 | ✅ 2 段分类器 | ✅ 智能审批 | — | — |
| Hook 体系 | ✅✅ 全事件 | gateway/shell/verify hook | plugin hook | before/after ToolCall |
| 威胁/依赖扫描 | — | ✅ threat/osv | — | 供应链加固 |
| 子智能体权限 | coordinator/swarm handler | ✅ auto-deny 默认 | 派生 + 收紧 | 继承 |

### 10.3 优劣分析

- **claude-code 的审批竞速** 最灵活：一个 ask 可被本地/IDE/手机/hook/分类器任一路解决，UX 与自动化兼顾。
- **hermes 的安全深度最硬核**：`HARD_BLOCKED` 先于 YOLO、`frozen YOLO` 防注入、`contextvar` 修真实 CVE —— 这三点是"被攻击过并加固过"的实战设计，最值得抄。
- **opencode 的 reject-反馈** 巧妙：拒绝不是死路，而是把理由回喂模型纠偏。
- **pi 无命令权限** 是有意的极简（靠 trust + 外置容器），但对不受信任任务风险高。

### 10.4 对 uAgentCli 的启示

安全栈建议（组合最优）：**hermes 两级 blocklist（HARD_BLOCKED 先行）+ frozen 危险模式 + contextvar 会话隔离** 作硬底线；**claude-code 竞速审批 + 来源分级规则 + 可选 LLM 分类器** 作决策层；**opencode reject-反馈** 提升 UX；**全事件 hook 体系** 供扩展；子智能体默认 auto-deny。

### 10.5 uAgentCli 落地细化设计

> 目标：**硬底线（blocklist）→ 分层规则 → 竞速审批 → 可选分类器**四层决策，全事件 hook 可扩展。三处 hermes 实战加固必抄。

**① 硬底线：两级 blocklist（借 hermes approval.py:519/595）**
```ts
const HARD_BLOCKED = [/* 绝不运行，【先于任何 bypass/YOLO】 */]   // rm -rf /、fork bomb…
```

**② 会话隔离修 CVE（借 hermes contextvar，GHSA-96vc-wcxf-jjff）**

```ts
const approvalCtx = new AsyncLocalStorage<ApprovalState>()   // 等价 Python contextvars
```

**③ 分层规则（借 opencode last-wins + cc 来源分级）**
```ts

//   defaults → userSettings → projectSettings → policy → cliArg → session
// evaluate 取【最后匹配规则】(last-wins)，确定性叠加（借 opencode）
```

**④ 竞速审批（借 cc interactiveHandler.ts:57，最灵活）**

**⑤ reject 反馈 + LLM 分类器（借 opencode + cc）**

```ts
// 可选 2 段 LLM 安全分类器(fast→thinking 评估整条 transcript)自动放行安全命令(借 cc auto 模式)
```

**⑥ 全事件 hook 体系（借 cc，供扩展/安全钩子）**
```ts

// 来自 settings + skill/agent frontmatter；Stop-hook 可阻止终止；SSRF guard 护 HTTP hook
// 子智能体默认 auto-deny（借 hermes _subagent_auto_deny）；威胁/依赖扫描 osv/threat（阶段4）
```

**⑦ 落地取舍**：**MVP** = HARD_BLOCKED + DANGEROUS_PATTERNS + frozen YOLO + last-wins 规则 + 本地 ask（对齐可用安全底线）；**阶段 2** = AsyncLocalStorage 会话隔离 + before/after ToolCall hook；**阶段 3** = 竞速审批（含 IM）+ reject 反馈；**阶段 4** = LLM 分类器 + 全事件 hook + 依赖/威胁扫描。**三处 hermes 加固（HARD_BLOCKED 先行、frozen YOLO、会话隔离）是被攻击过的实战设计，MVP 即应内建**。

## 11. 文件系统与持久化设计（uAgentCli 落地）

> 本章把散落在"记忆 / 上下文 / 技能 / 心跳 / 安全"各维度的**磁盘文件**统一成一套**可读、可编辑、可 diff、可同步**的文件系统契约。核心原则（借四家共识）：**结构化 markdown + 全文检索，不上向量库**；**索引常驻 context、正文按需加载**；**人类与智能体共写、archive-only 不删**。

### 11.1 目录总布局

```
# ① 全局（跨项目，~/.uagent/）
~/.uagent/
  config.json                 # 全局配置（模型/权限模式/默认渠道…）
  AGENT.md                    # 全局 agent 指令（人格/默认行为/全局约定）
  USER.md                     # 全局用户档案（"谁在用我"：角色/偏好/沟通风格）
  trust.json                  # 项目信任表（借 pi trust-manager）
  scheduled_tasks.json        # durable cron（借 cc，见 8.5）
  skills/                     # 用户级技能（SKILL.md，见 4.5）
  state.db                    # SQLite：sessions 元数据 + FTS5 索引（跨会话检索，借 hermes）
  projects/<project-slug>/    # 每个项目派生一个隔离目录（按项目路径 hash）
    memory/
      MEMORY.md               # 记忆【索引】——始终入 context（≤200 行/25KB，借 cc）
      <topic>.md              # 记忆【条目】——按需召回（frontmatter type，借 cc）
    sessions/
      <session-id>.jsonl      # 会话【事件树】——可 resume/fork（借 pi）
    USER.md                   # 项目级用户档案覆盖（可选）

# ② 项目内（随仓库走，团队共享部分入 git）
<repo>/
  AGENT.md                    # 项目 agent 指令（团队共享，入 git；兼容 AGENTS.md/CLAUDE.md）
  .uagent/
    config.json               # 项目配置（覆盖全局）
    AGENT.md                  # 项目私有覆盖（可 gitignore）
    settings.json             # 权限规则（denyWrite 保护对象之一，见 9.5/10.5）
    skills/                   # 项目技能
```

**全局 vs 项目 的分工**：`~/.uagent/` 存**跨项目、含隐私**的东西（用户档案、记忆、会话、cron）；`<repo>/` 存**随代码走、可团队共享**的东西（AGENT.md 指令、项目技能、权限规则）。记忆与会话**故意放全局 projects/**，不污染仓库、不误提交隐私。

### 11.2 五类核心文件的模块设计

**① `AGENT.md`（项目/智能体指令）—— 决定"智能体怎么做事"**
- 角色：system-prompt 注入的**行为/人格/工程约定**（代码风格、测试命令、目录含义…）。
- 兼容别名：`AGENT.md` / `AGENTS.md` / `CLAUDE.md`（借 opencode/pi 的候选列表，降低迁移成本）。
- **加载优先级 + 就近覆盖（借 opencode instruction.ts + pi resource-loader）**：
  ```
  全局 ~/.uagent/AGENT.md
    → 项目根 AGENT.md
    → 从"被触碰文件"逐级向上找嵌套 AGENT.md
  【第一条项目级匹配胜出，不逐层堆叠】（借 opencode：避免把每层祖先的 AGENT.md 全塞进 prompt）
  ```
- 支持 `@import ./path.md` 与 **http(s) URL 指令**（借 opencode，可引用团队共享规范）；每消息去重。

**② `USER.md`（用户档案）—— 决定"智能体为谁服务"**
- 与 AGENT.md 正交：AGENT.md 讲**项目**，USER.md 讲**人**（角色、专长、偏好、沟通风格、时区）。
- 来源：把 cc 的 `type: user` 记忆**提升为一等常驻文件**，始终入 context（不必等召回）。
- 全局 `~/.uagent/USER.md` + 项目级 `projects/<slug>/USER.md` 覆盖。
- 人类可直接编辑；智能体经"记忆 nudge"增量补写（append + archive-only，不覆盖人写内容）。
  ```markdown
  # USER
  role: 后端工程师，主 Go/Rust
  prefers: 简洁直给、先给结论；命令用 fish；PR 要中文描述
  timezone: Asia/Shanghai
  ```

**③ `MEMORY.md`（记忆索引）—— 常驻、低成本**
- ≤200 行 / 25KB，**始终入 context**；每条一行指针 `- [标题](topic.md) — 一句话钩子`（借 cc）。
- **两步保存协议**：先写 `<topic>.md`，再往 MEMORY.md 加一行指针。超限则由智能体合并/归档旧条目。

**④ `<topic>.md`（记忆条目）—— 按需召回、结构化**
```markdown
---
name: deploy-flow
description: 部署走 GitHub Actions，禁止手动 scp   # 供召回相关性判定
metadata: { type: user | feedback | project | reference }
---
正文（feedback/project 追加 **Why:** 与 **How to apply:** 两行）
关联用 [[other-topic]] 互链
```
- `type` 四类：`user`(人)/`feedback`(工作方式纠偏)/`project`(项目约束)/`reference`(外部链接)。
- **召回**：`startRelevantMemoryPrefetch` 每轮后台按 description 预取相关条目（借 cc）；跨会话则走 state.db FTS5（见下）。
- **archive-only**：过时条目移到 `memory/_archive/`，**从不物理删除**（借 hermes Curator 理念）。

**⑤ Session 文件（会话持久化）—— 双层：JSONL 真源 + SQLite 索引**
- **真源 = 追加式 JSONL 事件树（借 pi）**：每行一个事件（user/assistant/tool.call/tool.result/compaction…），带 `id` 与 `parentId`；`parentId` 让会话可 **fork / branch / resume**（配合 5.5 的会话树）。人类可读、可 grep、崩溃不坏历史。
  ```jsonl
  {"id":"e1","parentId":null,"type":"user","text":"..."}
  {"id":"e2","parentId":"e1","type":"assistant","text":"...","toolCalls":[...]}
  {"id":"e3","parentId":"e2","type":"tool.result","toolId":"...","result":"..."}
  ```
- **索引层 = `state.db`（SQLite + FTS5，借 hermes）**：会话落盘时同步写入 FTS5，供**跨会话全文检索**（DISCOVERY/SCROLL/BROWSE 三模式，零 LLM 成本，见 6.5）。JSONL 丢了能从 state.db 重建元数据，state.db 丢了能从 JSONL 重放重建——**双向可恢复**。
  ```sql
  CREATE TABLE sessions(id TEXT PRIMARY KEY, project TEXT, title TEXT, started_at INTEGER, ...);
  CREATE VIRTUAL TABLE msg_fts USING fts5(session_id, role, content, ts);
  ```

### 11.3 文件加载进 prompt 的顺序（每轮前置，memoized）

```
[system]  全局AGENT.md → 项目AGENT.md(就近唯一) → USER.md(全局+项目)
[memory]  MEMORY.md 索引（常驻） + 本轮预取的相关 <topic>.md
[context] 上一轮压缩后的会话历史（见 5.5）
[user]    当前用户消息（+ 触碰文件触发的嵌套 AGENT.md / 条件技能）
```
静态段（AGENT.md/USER.md/MEMORY.md）**memoized 缓存**，仅变更时失效，稳住 prompt-cache（借 cc）。

### 11.4 读写责任与保护

| 文件 | 人写 | 智能体写 | 入 git | 常驻 context | 保护（denyWrite） |
|---|:---:|:---:|:---:|:---:|:---:|
| `AGENT.md`(项目) | ✅ | ⚠️经审批 | ✅ | ✅ | — |
| `USER.md` | ✅ | ✅append | ❌(隐私) | ✅ | — |
| `MEMORY.md` | ✅ | ✅ | ❌ | ✅ | — |
| `<topic>.md` | ✅ | ✅ | ❌ | 按需 | — |
| `sessions/*.jsonl` | ❌ | ✅ | ❌ | 压缩后 | — |
| `state.db` | ❌ | ✅ | ❌ | — | — |
| `.uagent/settings.json` | ✅ | ❌ | ✅ | — | ✅ **无条件 denyWrite**（防配置逃逸，借 cc） |
| `trust.json`/`scheduled_tasks.json` | ✅ | ⚠️专用工具 | ❌ | — | ✅ 沙盒内 denyWrite |

**逃逸加固（借 cc，见 9.5）**：`settings.json`、`.uagent/skills`、bare-git 文件在沙盒内**无条件 denyWrite**，防止智能体经配置/技能注入或未沙盒 git 逃逸。

### 11.5 生命周期、GC 与同步

- **记忆 GC**：MEMORY.md 超 200 行触发智能体 consolidate（合并近义、归档过时到 `_archive/`），archive-only。
- **会话 GC**：JSONL 保留全量（可离线归档）；state.db 定期 `VACUUM`；上下文层用 5.5 的 prune 只清 prompt 内老工具输出，**不删磁盘**。
- **团队/多端同步（阶段 4，借 cc team memory）**：`AGENT.md`+项目技能随 git 自然共享；`USER.md`+`memory/` 可选走私有 git 仓或对象存储同步，实现"多端同一记忆/档案"。**隐私边界**：同步前对 `type:user/feedback` 走 `redact` 脱敏（借 hermes secret_scope/redact）。

### 11.6 落地取舍（MVP → 后置）

- **MVP**：`AGENT.md`(就近唯一加载) + `USER.md`(常驻) + `MEMORY.md`+`<topic>.md`(两步保存) + `sessions/*.jsonl`(可 resume)。纯文件，无 DB。
- **阶段 2**：`state.db` + FTS5 跨会话检索 + 相关记忆预取 + 记忆 nudge + JSONL↔DB 双向恢复。
- **阶段 4**：`_archive` 归档 + Curator consolidate + 多端同步 + 脱敏 + 全量逃逸加固。

## 12. 扩展机制深度分析：插件 / MCP / fork / hook（uAgentCli 落地）

> 这四者是"内核小、能力可扩"的**统一扩展面**：**插件**是能力载体、**MCP**是外部工具/资源接入、**hook**是生命周期切面、**fork**是执行隔离。uAgentCli 若走"小内核 + 插件化"路线（见第 1/12 章选型），这一章是地基。

### 12.1 插件 / 扩展系统（Plugin / Extension）

**四家范式（已核实）**

- **pi —— `ExtensionAPI`（第三方 TS 模块 hook 进内核，最彻底）**：`core/extensions/types.ts`（~1600 行）暴露 `registerTool` / `registerCommand` / `registerShortcut` / `registerFlag` / `registerMessageRenderer` / `registerEntryRenderer` / `registerProvider`(自定义 LLM provider) / `unregisterProvider` / `sendMessage`（向 UI/会话注入消息）；事件订阅 `on("session_compact", handler)`；**会话替换** `setup(sessionManager)` / `withSession(ctx)`；最关键的 **`BashOperations` seam**（`tools/bash.ts:56`）让扩展**重定向所有 bash 执行**（沙盒/远程 VM 就是这样接入的）。内核 7 工具，其余全靠 `pi.registerTool(...)`。
- **opencode —— `Plugin` 工厂返回 `Hooks`（函数式切面）**：`Plugin = (input: PluginInput, options?) => Promise<Hooks>`（`packages/plugin/src/index.ts:74`）。一个插件就是"注入 ctx → 返回一组 hook 回调"，hook 见 12.4；另可通过 `tool` 包定义自定义工具、`auth`/`provider` hook 扩展鉴权与模型。
- **claude-code —— 插件 + 技能 + MCP 三态**：`loadPluginHooks()`（`setup.ts:326`）预载插件 hook（`processSessionStartHooks` 消费）；能力扩展主要走**技能**（skill=command，见 4.5）与 **MCP**，插件偏 hook 切面。
- **hermes —— 目录式插件（约定优于配置）**：`plugins/context_engine/<name>/`（上下文引擎）、`plugins/platforms/`（渠道适配器）、外部 `MemoryProvider`（同一时刻仅一个，防 schema 膨胀）。**import 即注册**（见 3.5）本身就是工具插件化。

**对比表**

| 维度 | claude-code | hermes | opencode | pi |
|---|---|---|---|---|
| 扩展形态 | 插件 hook + 技能 + MCP | 目录式插件（约定） | `Plugin→Hooks` 工厂 | ✅ **`ExtensionAPI` register\*** |
| 加工具 | 技能/MCP | import 即注册 | `tool` 包 | ✅ `registerTool` |
| 加命令/UI | 技能/slash | — | — | ✅ command/shortcut/renderer |
| 换 LLM provider | fallback 链 | provider 无关 | `provider` hook | ✅ `registerProvider` |
| 执行重定向 seam | 沙盒 adapter | `BaseEnvironment` | — | ✅ **`BashOperations`** |
| 会话注入 | teammate 消息 | gateway | `client` | ✅ `sendMessage` |

**uAgentCli 落地设计**

### 12.2 MCP 服务连接（Model Context Protocol）

**四家能力（已核实）**

- **claude-code（最全）**：三 transport —— **stdio / SSE / Streamable HTTP** + 完整 **OAuth**（`services/mcp/oauthPort.ts` 本地回调端口、`xaa.ts` 走 IETF **ID-JAG token-exchange** `urn:ietf:params:oauth:grant-type:token-exchange`、`claudeai.ts` 官方登录）；**in-process SDK MCP**（`SdkControlTransport.ts`：MCP server 跑在进程内，不 spawn 子进程）；**Elicitation**（`elicitationHandler.ts` MCP server 反向向用户要输入）；**channel 权限/allowlist**（`channelPermissions/channelAllowlist`）+ 审批 Dialog。MCP 工具并入同一工具池与权限管线。
- **opencode（最全）**：`src/mcp/` 有 `oauth-provider.ts`（`/mcp/oauth/callback`）、`oauth-callback.ts`、`auth.ts`（持久 `oauthState`）；`config/mcp.ts` 配置；stdio + Streamable HTTP + OAuth；测试含 **session-recovery**（连接断了能恢复）。

- **pi（弱/缺）**：无 `mcp/` 目录，MCP 非一等公民，需靠扩展自行接。

### 12.3 fork 机制（执行隔离与分叉）

> "fork"在四家有**五种含义**，用途不同，勿混：

| fork 种类 | 是什么 | 代表实现 | 用途 |
|---|---|---|---|
| **技能 fork** | 技能起隔离子智能体执行（独立 context + token 预算） | cc `context: 'fork'`（`command.ts:45`、`loadSkillsDir.ts:260`） | 自包含技能不污染主上下文 |
| **子智能体 fork** | Task/delegate 起子 agent | cc AgentTool、opencode `subagent_type`（`prompt.ts:295`）、hermes `delegate_task` | 分解任务、并行 |
| **会话 fork/branch** | 从某历史条目分叉出新会话线 | pi `session.fork(entryId)` + `session_before_fork` hook（`agent-session-runtime.ts:259`）、cc `forkSession`（`agentSdkTypes.ts:268`） | 探索式回溯、A/B 试不同走向 |
| **压缩 fork** | 起 compact 子智能体做全量摘要 | cc autocompact fork、opencode `bridge.fork`（`llm.ts:282`） | 摘要不占主循环 |
| **worktree fork** | 建隔离 git worktree | cc `EnterWorktreeTool`（`.uagent/worktrees/`） | 并行改文件不踩踏（见 9.5） |

**共性设计**：fork 出的执行体**隔离 `agentId` + 文件状态缓存 + token 预算**，只把**结果**回传父级（借 cc）；会话 fork 靠 JSONL 的 `parentId`（见 11.2）天然支持。

**uAgentCli 落地设计**
```ts
export interface ForkOptions {
  kind: 'skill' | 'subagent' | 'session' | 'compaction' | 'worktree'
  isolateContext: boolean          // 独立消息历史
  tokenBudget?: number             // 独立预算（借 cc task_budget）
  fromEntryId?: string             // 会话 fork：JSONL parentId 锚点（见 11.2）
  worktreeDir?: string             // worktree fork（见 9.5）
}
// 统一入口，返回只回传结果的隔离执行句柄；session_before_fork 可被 hook 拦截（借 pi）
fork(opts: ForkOptions): Promise<ForkHandle>
```
取舍：**MVP** = 子智能体 fork（Task 工具）+ 技能 fork；**阶段2** = 压缩 fork + worktree fork；**阶段4** = 会话 fork/branch（配合 5.5 会话树）。

### 12.4 hook 机制（生命周期切面）

**claude-code（最全，全事件体系）**——事件（已核实，`schemas/hooks.ts`）：
`PreToolUse` / `PostToolUse` / `PostToolUseFailure` / `UserPromptSubmit` / `SessionStart` / `SubagentStart` / `SubagentStop` / `Stop` / `PreCompact` / `PermissionRequest` / `PermissionDenied` / `Notification` / `FileChanged` / `CwdChanged` / `WorktreeCreate` / `Elicitation` / `Setup`。
关键语义：**`Stop`-hook 可阻止终止**（`stopHooks.ts:271` 返回 `preventContinuation`/`stopReason` 逼再干一轮）；来源 = settings + skill/agent frontmatter；**SSRF guard 护 HTTP hook**。

**opencode（函数式 Hooks 接口，已核实 `plugin/src/index.ts:222`）**：
`event` / `chat.message` / `chat.params` / `chat.headers` / `permission.ask` / `command.execute.before` / `tool.execute.before` / `tool.execute.after` / `shell.env` / `tool.definition`(改工具描述/参数) / `experimental.chat.messages.transform` / `experimental.chat.system.transform` / `experimental.session.compacting` / `experimental.compaction.autocontinue` / `experimental.provider.small_model` / `experimental.text.complete`。**"input/output 双参"约定**：hook 可**原地改写 output** 影响后续（如改 system prompt、改工具参数、改 permission 结果）。

**pi**：`agent.beforeToolCall({toolCall,args})`（可 block）/ `agent.afterToolCall({result,isError})`（可改写/终止）（`agent-session.ts:424/445`）+ `session_before_fork` 等生命周期。

**hermes**：gateway/shell/verify + 插件生命周期 hook（compression/plugin reset per-session）。

**对比表**

| 维度 | claude-code | hermes | opencode | pi |
|---|---|---|---|---|
| 事件数 | ✅✅ ~17 全事件 | gateway/shell/verify | ✅ ~16 函数式 | before/after + fork |
| 改写能力 | 决策/注入 | 审批/校验 | ✅ **双参原地改 output** | ✅ 改 result |
| 阻止终止 | ✅ **Stop-hook** | verify-on-stop | compaction.autocontinue | — |
| 改工具定义 | — | check_fn | ✅ `tool.definition` | — |
| 改 system/消息 | frontmatter | 压缩 hook | ✅ transform | prompt 选项 |
| 来源 | settings+frontmatter | 插件 | 插件 | 扩展 |
| 安全护栏 | ✅ SSRF guard | — | — | — |

**uAgentCli 落地设计**
```ts
// 统一 hook 总线：借 opencode 的"双参原地改写" + cc 的"全事件 + Stop 可阻止"
export type HookEvent =
  | 'SessionStart' | 'UserPromptSubmit'
  | 'PreToolUse' | 'PostToolUse' | 'PostToolUseFailure'   // 借 cc
  | 'PermissionRequest'                                   // 竞速审批一路（见 10.5）
  | 'PreCompact' | 'SessionStop' | 'SubagentStart' | 'SubagentStop'
  | 'ToolDefinition'                                       // 借 opencode：改描述/参数
export type HookFn<E> = (input: HookInput<E>, output: HookOutput<E>) => Promise<void>
// 语义：hook 原地改 output → 影响后续；PreToolUse 可 deny；SessionStop 可 preventStop
// 护栏：HTTP hook 走 SSRF guard（借 cc）；子智能体 hook 默认收紧
registerHook<E extends HookEvent>(event: E, fn: HookFn<E>): Disposable
```
取舍：**MVP** = `PreToolUse`(可 deny) + `PostToolUse` + `SessionStart`（对齐 pi before/after）；**阶段2** = `SessionStop` 阻止终止 + `PreCompact` + `ToolDefinition` 改写；**阶段4** = 全事件 + SSRF guard + frontmatter 来源。**PreToolUse 与权限管线是同一切面**——统一到 10.5 的竞速审批，勿两套。

### 12.5 四者的关系与统一扩展面

```
              ┌──────────── ExtensionAPI（唯一扩展面, 借 pi）────────────┐
   插件 Plugin │ registerTool/Channel/ContextEngine/Provider + on(hook) │
（能力载体）   └───────┬───────────────┬───────────────┬────────────────┘
                      │               │               │
                 registerTool     on(hook)      setBashOperations
                      │               │               │
                 ┌────▼────┐     ┌────▼────┐     ┌─────▼─────┐
                 │  MCP    │     │  Hook   │     │  Fork /   │
                 │(外部工具)│     │(生命周期 │     │  Sandbox  │
                 │→工具池   │     │ 切面)   │     │ (执行隔离) │
                 └─────────┘     └─────────┘     └───────────┘
   MCP 工具经 ToolDef 并入同一池+权限管线；hook 的 PreToolUse 即权限切面；
   fork 隔离执行体经 BashOperations seam 落到沙盒后端。四者共用一套 ToolDef/权限/事件契约。
```

**一句话**：uAgentCli 只需**一个 `ExtensionAPI` + 一套 `ToolDef` 契约 + 一条权限管线 + 一个 hook 总线**，插件/MCP/fork/hook 就都挂在其上——这是"小内核可扩展"的关键，避免四套并行机制。

## 13. Prompt 设计与上下文注入策略（uAgentCli 落地）

> System prompt 不是一坨字符串，而是**分层组装 + 按变动频率排序 + 缓存友好 + 按模型/模式差异化**的工程。四家在此差异极大，hermes 的文档最系统、opencode 的按模型分家最工程化、cc 的 memoize+ephemeral 最精细。

### 13.1 System prompt 的分层结构（四家对比，已核实）

- **hermes（分层最清晰）**——`agent/system_prompt.py` 把缓存 system prompt 组装成**三个有序 tier**（`website/docs/developer-guide/prompt-assembly.md` 权威文档）：
  1. **stable（最不常变）**：identity（`SOUL.md` 或 fallback）→ tool/model 行为指引 → skills 索引 prompt → environment hints → platform hints；
  2. **context（偶变）**：调用方 `system_message` + 项目上下文文件（`.hermes.md`/`AGENTS.md`/`CLAUDE.md`/`.cursorrules`）；
  3. **volatile（每轮变）**：`MEMORY.md` 快照 + `USER.md` 用户档案 + 外部 memory-provider 块 + `timestamp/session/model/provider` 行。
  **最终顺序 `stable → context → volatile`**——这个顺序就是为缓存服务（见 13.3）；**注意：skills 属 stable、memory/profile 属 volatile，但都在缓存 system prompt 内，不是临时中途 overlay**。
- **claude-code（memoize + 段化）**：`getSystemContext`（env/gitStatus/目录结构）与 `getUserContext`（CLAUDE.md + 嵌套 memory）均 **`memoize`**（`context.ts:116/155`），仅变更时 `cache.clear()`。系统提示由**命名 section** 组成，每段一个 `ComputeFn`（`systemPromptSections.ts`），**返回 null 即跳过**（按需装配）。自主体身份还含 memory + env + **proactive section**。
- **opencode（按模型分家）**：`session/system.ts` 按**模型家族选不同 prompt 文件**——`anthropic.txt`/`gpt.txt`/`codex.txt`/`gemini.txt`/`kimi.txt`/`meta.txt`/`beast.txt`/`default.txt`；`environment()` 拼 `<env>` 块 + `<available_references>`（项目指令文件清单）+ `<mcp_instructions>`。另有**模式 prompt**（`plan-mode.txt`/`plan-reminder-anthropic.txt`）与**子智能体 prompt**（`compaction.txt`/`explore.txt`/`summary.txt`/`title.txt`）。
- **pi（XML 注入 + 可覆盖）**：`core/system-prompt.ts` 拼 `<project_context>…</project_context>`（项目上下文文件）+ `<available_skills>` XML（`<skill><name><description><location>`，让模型**自己用 read 工具读技能**，`system-prompt.ts:12`）+ `appendSystemPrompt` 追加段 + `SYSTEM.md`/`APPEND_SYSTEM.md` 覆盖/追加。

**对比表**

| 维度 | claude-code | hermes | opencode | pi |
|---|---|---|---|---|
| 组织方式 | 命名 section + ComputeFn | ✅ **3-tier 按变动频率** | 模型家族 .txt + 拼装 | XML 段拼接 |
| 身份来源 | 自主体 identity | `SOUL.md`/fallback | 各模型 .txt | 内置 + SYSTEM.md |
| 项目指令注入 | CLAUDE.md+嵌套 | .hermes.md/AGENTS/CLAUDE | `<available_references>` | `<project_context>` |
| 技能注入 | 动态发现 | stable 层 skills 索引 | 工具描述 | ✅ `<available_skills>` XML |
| 按模型差异化 | output style | tool-use 强制(GPT/Codex) | ✅✅ **8 套 .txt 分家** | provider 无关 |
| 按需/跳过 | ✅ null section | tier 缺则省 | 条件拼 | 条件段 |
| 用户可覆盖 | output style | system_message | agent.prompt | ✅ SYSTEM/APPEND_SYSTEM.md |

### 13.2 上下文注入：注入什么 / 何时 / 注入到哪

**"注入什么"× "在哪一层"× "多久变一次"**（综合四家，映射到 uAgentCli 第 11 章文件）：

| 注入项 | 归属层 | 注入位置 | 变动频率 | 来源文件 |
|---|---|---|---|---|
| Agent 身份/人格 | stable | system 头部 | 几乎不变 | `AGENT.md`/SOUL |
| 工具/模型行为指引 | stable | system | 版本级 | 内置（按模型选） |
| 技能索引 `<available_skills>` | stable | system | 技能增删时 | `skills/*/SKILL.md` |
| environment（os/cwd/git） | stable/context | system `<env>` | 会话级 | 运行时探测 |
| 项目指令 | context | system | 偶变 | `AGENT.md`（就近唯一，见 11.2） |
| 调用方 system_message | context | system | 会话级 | config/API |
| MEMORY.md 索引 | **volatile** | system 尾部 | 每轮可变 | `MEMORY.md`（见 11.2） |
| USER.md 用户档案 | **volatile** | system 尾部 | 偶变 | `USER.md` |
| 相关记忆预取块 | volatile | system 尾/user 前 | 每轮 | 预取 `<topic>.md` |
| timestamp/session/model | volatile | system 尾部 | 每轮 | 运行时 |
| 会话历史（压缩后） | — | messages | 每轮 | session JSONL（见 5.5） |
| 触碰文件的嵌套 AGENT.md / 条件技能 | — | user 消息 | 触发时 | 就近发现（见 4.5/11.3） |
| 工具结果 | — | tool message | 每次 | 运行时 |

**三条注入原则（四家共识）**：
1. **静态段前置、易变段后置**——system prompt 内也按 stable→volatile 排（借 hermes）；
2. **"每轮都在"的东西放 system 里而非反复塞 user**——memory 是每轮注入的常驻块（借 hermes：`Memory is injected into every turn`），不是临时 overlay；
3. **就近触发的东西放 user 侧**——嵌套 AGENT.md / 条件技能随"被触碰文件"在**当轮 user 消息**注入，不污染缓存 system（借 cc/opencode）。

### 13.3 Prompt cache 稳定性策略（关键，四家都重视）

- **按变动频率排序**（借 hermes stable→context→volatile）：把最不变的放最前，缓存前缀命中最长；一旦把 timestamp 放前面，整段缓存作废。
- **memoize 静态段**（借 cc）：`getSystemContext`/`getUserContext` 缓存，仅底层文件变更才 `clear`，避免每轮重算重排。
- **cache_control 断点**（借 hermes `prompt_caching.py`：单布局 `system_and_3` = **4 个 cache_control breakpoint**——system + 3 处）：显式打断点让 provider 缓存到断点；`ephemeral` 类型（借 cc `api.ts:73`）。
- **provider 感知的 marker 放置**（借 hermes）：native Anthropic 顶层 marker；OpenRouter 对 `role:tool` 顶层 marker 静默拒绝 → 放到 content part；空 assistant（纯 tool_calls）特殊处理。**这是多 provider 必踩的坑**。
- **工具池分区稳定排序**（借 cc `assembleToolPool`，见 3.5）：合并 MCP 工具时各分区独立排序，保持工具列表字节稳定，护住缓存。

### 13.4 按模型 / 模式差异化

- **按模型分家**（借 opencode，最工程化）：不同模型家族用不同 system prompt（Anthropic/GPT/Gemini/Codex/Kimi/…），因各家对指令风格、工具调用格式、思维标签的偏好不同。
- **tool-use 强制段**（借 hermes）：仅对 GPT/Codex 类模型注入 `You MUST use your tools to take action` —— 弥补部分模型"爱描述不爱动手"。
- **模式 prompt**（借 opencode）：`plan-mode` 注入只读约束 + `plan-reminder` 周期提醒；`build/explore` 各有 ruleset（见 4.1）。
- **子智能体专用 prompt**（借 opencode）：compaction/explore/summary/title 各有精简 prompt，不复用主 prompt。

## 14. 多智能体策略深度分析（uAgentCli 落地）

> 多智能体不是"多开几个 loop"，而是**预制子智能体 + 主-子通信 + 上下文隔离/继承 + 工具池策略**四件事的组合。四家从"完全外置"（pi）到"工业级编排"（cc coordinator/swarm）跨度极大。

### 14.1 四家范式总览

- **claude-code（最全，四种协作模型并存）**：`AgentTool` 递归 + `builtInAgents`（一次性 worker，跑完回一份 report，父**永不** SendMessage 续之，`AgentTool/constants.ts`）；**隐式 fork 子智能体**（省略 `subagent_type` → 子**继承父完整对话上下文 + system prompt + 精确工具池**，`tools:['*']`+`useExactTools` 做到 cache-identical，`forkSubagent.ts`）；**coordinator 模式**（`coordinatorMode.ts`，主体只做编排+对用户说话）；**swarm/teammate**（`SendMessage` 双向 + scratchpad 共享）。用 **AsyncLocalStorage** 隔离每个 agent 的上下文（`agentContext.ts`，并发安全）。
- **opencode（agent = 权限 ruleset）**：内置 `build`/`plan`（`mode: primary`）+ `general`/`explore`（`mode: subagent`）+ `compaction`/`title`/`summary`（`hidden:true` 内部）（`agent/agent.ts`）；`task` 工具 spawn **带 `parentID` 的子 session**（`prompt.ts:267-304`）；`Agent.generate` 用 LLM **动态生成** agent 配置。
- **hermes（delegate + MoA + Kanban 三层）**：`delegate_task` spawn 子 `AIAgent`（**隔离 context/无父历史 + 受限 toolset + 聚焦 system prompt**，父**阻塞**到子完成、只见 summary，`MAX_DEPTH=1`，`delegate_tool.py`）；**MoA**（Mixture-of-Agents：多个 reference advisor 用不同模型/provider 把"最佳思考"喂给 aggregator，aggregator 独占工具调用，`moa_loop.py`）；**Kanban swarm**（root→并列 specialist worker→verifier + blackboard 共享，`kanban_swarm.py`）。
- **pi（完全外置）**：**loop 内无 Task 工具**；多智能体全在 `packages/orchestrator/` —— supervisor `spawnInstance` 起 pi 子进程，`orchestrator spawn/rpc/rpc-stream` 走 **JSONL RPC**（`handler.ts:59`、`cli.ts`）；Radius 做分布式 presence。

**对比表**

| 维度 | claude-code | hermes | opencode | pi |
|---|---|---|---|---|
| 子智能体入口 | ✅ AgentTool + fork + coordinator | ✅ delegate + MoA + Kanban | ✅ task 工具（子 session） | ⚠️ **外置 orchestrator** |
| 预制子智能体 | ✅ builtInAgents 一次性 | delegate 目标构建 | ✅ build/plan/general/explore/隐藏 | ❌ 无内置 |
| 通信模型 | ✅✅ **4 种并存** | 3 种（阻塞/MoA/看板） | 子 session report | RPC |
| 上下文 | 隔离 / **fork 继承全量** | ✅ 隔离(无父史) | 子 session parentID | 子进程独立 |
| 并发隔离 | ✅ AsyncLocalStorage | ThreadPool per-worker | Effect fiber | 进程 |
| 工具池 | 受限 / fork 精确同池 | ✅ 受限+剥离危险 | ✅ agent=ruleset | 各自内置 |
| 编排角色 | ✅✅ coordinator | orchestrator profile | primary agent | 外置 |
| 深度上限 | 有 | ✅ MAX_DEPTH=1 | 子不再 spawn | — |

### 14.2 预制子智能体（preset subagents）

| 项目 | 预制角色 | 定义方式 | 特点 |
|---|---|---|---|
| **claude-code** | `builtInAgents`（一次性 worker）+ 隐式 fork agent | 静态注册 + frontmatter agent | worker 跑完回 report；fork agent **不注册**、仅 `!subagent_type` 时用 |
| **opencode** | `build`/`plan`(primary) `general`/`explore`(subagent) `compaction`/`title`/`summary`(hidden) | `agent.ts` 内置 + 文件自定义 + `Agent.generate` | 每个 = **权限 ruleset**：build 全开、plan 拒 edit、explore 近全 deny |
| **hermes** | 无固定"角色卡"，delegate 按目标现构 + MoA advisor/aggregator slot | `delegate_task(goal, toolset, ...)` + `moa` 配置 | 现场构建聚焦子体；MoA slot 可跨模型 |
| **pi** | 无内置（外置 orchestrator 决定） | orchestrator spawn | 内核零多智能体负担 |

**uAgentCli 取舍**：抄 **opencode 的"少量高质量预制角色 + agent=ruleset"**（`build`/`plan`/`explore`/`general`）作默认，配 **cc 的 `builtInAgents` 一次性 worker 语义**；长任务再引入 **hermes delegate 的现场构建**。预制角色用 frontmatter 定义（复用 4.5 技能格式），`mode: primary|subagent|hidden`。

### 14.3 主-子通信机制（四种模型，勿混用）

| 通信模型 | 语义 | 代表实现 | 适用 |
|---|---|---|---|
| **① 一次性 report** | 子跑完回单份结果，父不再续 | cc `builtInAgents`、opencode task 子 session、hermes delegate(阻塞) | 分解型子任务 |
| **② 异步 task-notification** | 结果作 **user-role 消息**（`<task-notification>` XML）异步到达父 | cc coordinator/fork（`coordinatorMode.ts:144-159`） | 并行 worker、远程 agent |
| **③ 双向 SendMessage** | teammate 间 `to:"name"` 点对点 / `to:"*"` 广播；**纯文本对队友不可见，必须用工具** | cc swarm（`teammatePromptAddendum.ts:12`） | 对等协作 team |
| **④ 共享黑板/看板** | 经共享文件/DB 解耦通信，durable 跨 worker | cc scratchpad（免审批读写）、hermes Kanban **blackboard** + board claim | 长期、多 worker、可断续 |
| **⑤ 外置 RPC** | 父经 JSON-RPC 驱动子进程 | pi orchestrator `rpc/rpc-stream` | 进程/机器级隔离 |

**关键洞察**（借 cc coordinator 提示词）：worker 结果是**内部信号不是对话方**——"never thank or acknowledge them"；主体每条输出都是**说给用户**的。这条规则避免主体把子体结果当成对话来回复、浪费轮次。

**uAgentCli 落地**
```ts
// packages/core/src/multiagent/channel.ts
export type CommModel =
  | { kind: 'report' }                               // ① 一次性（借 cc/opencode/hermes）
  | { kind: 'notification' }                          // ② 异步 <task-notification>（借 cc）
  | { kind: 'message'; to: string | '*' }             // ③ SendMessage 双向（借 cc swarm）
  | { kind: 'blackboard'; path: string }              // ④ 共享黑板/scratchpad（借 cc/hermes）
  | { kind: 'rpc' }                                   // ⑤ 外置进程（借 pi）
// 结果统一以【user-role + <task-notification> 包裹】回注父 loop（借 cc），主体靠 XML 标签识别
```

### 14.4 上下文机制（隔离 vs 继承）

- **隔离型（默认，借 hermes/cc subagent）**：子体**全新对话、无父历史**，只带"目标 + 精选 context"；父只见"委派调用 + summary 结果"（`delegate_tool.py:15`）。省 token、防污染、聚焦。
- **继承型（借 cc fork）**：子体**继承父完整对话 + system prompt + 精确工具池**（`forkSubagent.ts:24`），用于"换个身份接着干"（如 compaction/orchestration），且做到 **cache-identical**（同工具池同字节，命中父缓存）。
- **子 session 型（借 opencode）**：子是带 `parentID` 的真实 session（`prompt.ts:271`），可独立 resume，父子经 session 树关联（呼应 5.5/11.2 会话树）。
- **并发隔离（借 cc `agentContext.ts`）**：用 **AsyncLocalStorage** 让并发子体的上下文（analytics/审批/状态）互不串扰——**这是并发多智能体的正确姿势**（对照 10.5 hermes 用 contextvar 修 CVE，同一思想）。

```ts
export interface SpawnOptions {
  contextMode: 'isolated' | 'inherit' | 'subsession'   // 三态（借 hermes/cc/opencode）
  goal: string
  seedContext?: Message[]                               // isolated 时的精选上下文
  parentSessionId?: string                              // subsession/树关联
  tokenBudget?: number                                  // 独立预算（借 cc task_budget）
}
// 并发子体一律跑在 AsyncLocalStorage 隔离域内（借 cc）
```

### 14.5 工具机制（子智能体工具池）

- **受限工具池（隔离子体，借 hermes）**：子体拿**裁剪后的 toolset**，**危险工具无条件剥离** + **禁递归**（`delegate_task`/`cronjob` 从子体移除，防"子体再派活/再排期"，`delegate_tool.py:47-52`）。
- **精确同池（fork 子体，借 cc）**：fork 子体拿**父的精确工具池**（`tools:['*']`），为 cache-identical。
- **agent = 权限 ruleset（借 opencode）**：每个预制角色自带权限集——`build` 全开、`plan` 拒所有 edit、`explore` 近全 deny（`agent.ts`）；工具可见性按 deny 过滤。
- **审批回调注入（借 hermes 实战细节）**：子体 worker 线程经 `ThreadPoolExecutor(initializer=...)` 注入**非交互审批回调**，避免子体去抢父 TUI 的 stdin 造成死锁（`delegate_tool.py:61-67`）——多线程子体的必修坑。
- **子智能体默认 auto-deny（借 hermes，见 10.5）**：子体权限默认更严，敏感操作默认拒。

```ts
export interface SubagentToolPolicy {
  base: 'inherit-exact' | 'restricted'      // fork=inherit-exact；isolated=restricted（借 cc/hermes）
  strip: string[]                            // 无条件剥离（delegate/cron/危险工具）
  ruleset?: PermissionRule[]                 // agent=ruleset（借 opencode）
  autoDeny: boolean                          // 子体默认 auto-deny（借 hermes）
}
```

### 15.3 分阶段落地路线图

- **阶段 2（主动性）**：durable cron（文件锁 + jitter）+ 智能体自建 cron + 闲时/监控触发 + 类型化 memdir 记忆 + FTS5 跨会话检索 + 周期 nudge。→ 获得 claude-code/hermes 的"主动 7×24"能力。
- **阶段 3（多渠道）**：Client/Server + 生成式 SDK + IM Gateway（platform_registry 懒加载 + deliver + mirror）+ 审批渠道化。→ 对标 hermes 多渠道。
- **阶段 4（强隔离 + 自进化）**：可插拔沙盒后端（bubblewrap/Docker 加固 + 逃逸加固 + serverless）+ Curator 自进化技能 + frozen YOLO/contextvar 安全加固 + 全事件 hook 体系。→ 达到工业级。

### 16.2 术语表

- **run-loop**：智能体"调模型→执行工具→回填结果→再调模型"的主循环。
- **compaction / autocompact**：上下文超长时对历史消息做摘要压缩。
- **microcompact**：外科式丢弃个别旧工具输出（保粒度，不整体摘要）。
- **MCP（Model Context Protocol）**：标准化的工具/资源/提示扩展协议。
- **fork/inline（技能执行）**：inline 把技能提示展开进当前对话；fork 起隔离子智能体执行。
- **progressive disclosure（渐进披露）**：技能先给元数据，用时再按需加载正文/引用，省 token。
- **check_fn 门控**：按运行环境（OS/env/角色）动态决定工具是否进 schema。
- **durable cron**：持久化到磁盘、跨重启存活的定时任务。
- **bubblewrap / Seatbelt**：Linux / macOS 的进程级沙盒机制。
- **HARD_BLOCKED**：即便在 YOLO 模式也绝不执行的命令黑名单。
- **contextvar 会话隔离**：用 Python contextvars 让并发会话的审批状态互不串扰（修 CVE）。
- **steering**：用户在智能体运行中途插话/改向的机制。
- **AGENT.md / AGENTS.md**：项目级智能体指令文件（行为/约定），system-prompt 注入，团队共享入 git。
- **USER.md**：用户档案文件（角色/偏好/沟通风格），常驻 context，含隐私不入 git。
- **MEMORY.md**：记忆索引文件（一条一行指针），≤200 行/25KB，始终入 context。
- **topic 记忆（`<topic>.md`）**：单主题记忆条目，带 `type: user|feedback|project|reference` frontmatter，按需召回。
- **JSONL 会话树**：追加式、每行一事件、带 parentId 的会话真源，可 resume/fork。
- **state.db（FTS5）**：SQLite 全文检索索引，供跨会话零 LLM 成本召回；与 JSONL 双向可恢复。
- **archive-only**：过时记忆移入 `_archive/` 而非物理删除。
- **ExtensionAPI**：内核暴露给第三方模块的统一扩展面（register 工具/命令/渠道/provider + hook 订阅）。
- **BashOperations seam**：把所有 bash 执行抽成可替换接口，扩展据此把命令路由进沙盒/远程 VM。
- **in-process SDK MCP**：MCP server 跑在宿主进程内、不 spawn 子进程（借 cc SdkControlTransport）。
- **ID-JAG token-exchange**：MCP OAuth 用的 IETF 身份断言授权 grant，跨服务换 token。
- **fork（五种）**：技能 fork / 子智能体 fork / 会话 fork(branch) / 压缩 fork / worktree fork——统称隔离执行分叉。
- **hook（生命周期切面）**：在 PreToolUse/PostToolUse/SessionStop 等事件点注入回调，可改写 output 或阻止终止。
- **Stop-hook**：可在智能体欲终止时返回 preventContinuation 逼其再干一轮（借 cc）。
- **prompt 三层（stable/context/volatile）**：system prompt 按变动频率分层——身份/工具/技能(stable)→项目指令(context)→记忆/时间戳(volatile)，为缓存服务（借 hermes）。
- **cache_control / ephemeral**：在消息上打缓存断点让 provider 缓存前缀；`ephemeral` 是短时缓存类型。
- **provider 感知 marker**：不同 provider 对 cache_control 放置要求不同（Anthropic 顶层 vs OpenRouter content-part），需分别处理。
- **按模型分家 prompt**：不同模型家族用不同 system prompt 基座（Anthropic/GPT/Gemini/Codex…，借 opencode）。
- **`<available_skills>` / `<project_context>`**：注入技能索引 / 项目上下文的 XML 段（借 pi）。
- **预制子智能体（preset subagent）**：内置的固定角色子体（如 build/plan/explore），常自带权限 ruleset（借 opencode）。
- **fork 子智能体（继承型）**：子体继承父完整对话+system prompt+精确工具池，做到 cache-identical（借 cc）。
- **delegate（隔离型）**：子体全新对话、无父历史、受限工具，父只见 summary（借 hermes，MAX_DEPTH=1）。
- **coordinator 模式**：主体只做编排+对用户说话，worker 结果作内部信号不当对话方（借 cc）。
- **`<task-notification>`**：包裹子体结果、以 user-role 消息异步回注父 loop 的 XML 标签（借 cc）。
- **MoA（Mixture-of-Agents）**：多个 reference advisor（可跨模型）把最佳思考喂给独占工具调用的 aggregator（借 hermes）。
- **blackboard / Kanban swarm**：经共享黑板/看板解耦的多 worker 长任务编排（借 hermes）。
- **AsyncLocalStorage 隔离**：让并发子智能体的上下文/审批状态互不串扰（借 cc，与 contextvar 同思想）。
